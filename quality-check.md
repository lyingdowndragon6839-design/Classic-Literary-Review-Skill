# Quality Check

这个清单用于两件事：检查一篇读后感是否可交付，
也检查仓库本身是否像一个可维护的开源项目。

## 写作质量

- [ ] 文章有一个明确主判断。
- [ ] 剧情复述不超过全文 20%。
- [ ] 主要论点都有具体人物、情节、物件或叙述方式支撑。
- [ ] 用户自己的阅读感受没有被外部评论盖住。
- [ ] 文章没有用“人性、命运、意义”填补论证空白。
- [ ] 结尾回到具体场景、人物或问题，而不是万能口号。

## Humanization

- [ ] 已删除“首先、其次、最后、总而言之、由此可见”等模板连接词。
- [ ] 段落长度和句子节奏有变化。
- [ ] 至少有一个具体阅读过程或重读发现。
- [ ] 抽象词被场景、动作、物件替代。
- [ ] 修改没有抹掉用户原意。

## AI Trace

- [ ] “受益匪浅、发人深省、体现了、揭示了”等套话已改写。
- [ ] 没有空泛赞美。
- [ ] 没有机械过渡。
- [ ] 没有为深刻而深刻的句子。
- [ ] 文章不能只替换书名就套到另一部作品。

## Research Integrity

- [ ] 联网时有 Research Summary。
- [ ] 无法访问的来源标记为 blocked。
- [ ] 引用外部资料时区分观点和写法。
- [ ] 没有复制、改写或拼接外部文章。
- [ ] 离线模式没有声称“检索发现”。

## Repository Audit

- [ ] 文件名统一使用小写 kebab-case，GitHub 标准文件除外。
- [ ] 每个 Markdown 文件只有一个一级标题。
- [ ] 标题层级从 `#`、`##`、`###` 逐级展开。
- [ ] 没有空章节、空文件或单行压缩内容。
- [ ] 表格、列表、引用和代码块能正常渲染。
- [ ] Mermaid 代码块使用 `mermaid` 语言标记。
- [ ] README 链接到的文件全部存在。
- [ ] `examples/` 中每本书有独立文件。
- [ ] `modules/` 中每个模块各自成文。
- [ ] 重复内容只保留在必要模板里。
- [ ] 语言自然、克制，不像营销文案。

## Self-Check Commands

```powershell
git status --short
rg --files
rg -n "\]\(([^)]+)\)" -g "*.md"
```

最终提交前还要运行链接检查和文件命名检查。
